OSD Type

According to their forms and usages, OSDs are classified as follows :

Policy

If there is the overlay rule specifying specially in each IA specification, you have to apply the specifications above all.

In case it is not so, follow under common specifications.

Full Window

A full window is displayed on the whole screen on the top layer.
When a full window opens while an OSD is already being displayed, the previous OSD closes and only the new full window is displayed.
If the full screen originates from or is linked to the previous OSD, the UI should return to the previous OSD when the Return function is executed.

Parial Window
A partial window is overlaid over a source or a full window.
A partial window must not be overlaid over another partial OSD, such as partial window, popup, notice and control. Instead, the previous OSD closes and a new partial window opens.
Ongoing status information among notice will not close until the status changes.
If a partial window subordinate to an input source (such as Channel List, etc.) opens, the source is switched and the full window closes.
Popup
A popup usually originates from a full or partial window and is overlaid over the previous OSD.
A popup must not be overlaid over another notice and control. Instead, the previous OSD closes and a new popup opens.
When a new popup appear from the current popup, only the new one is displayed. When the new popup is closed, the previous popup is displayed.
Notice
A notice notifies users of a specific state or information as the first priority and therefore should be displayed on the top layer at any circumstance.
Notices are classified into normal event and ongoing status. Ongoing status includes pairing and mute.
Event notices will automatically timeout after a certain amount of time or immediately close when another function is executed.
When a new event appears, only the new one is displayed.
Ongoing status information will not close until the status changes and will remain even if another function is executed.
Control
Users use controls to control or set the previous OSD and therefore controls should be displayed on the top layer at any circumstance.
This is only if the previous OSD supports the control action using controls.
A control must not be overlaid over another notice and control. Instead, the previous OSD closes and a new control opens.
